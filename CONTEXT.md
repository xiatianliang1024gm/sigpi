# Context

Domain glossary for SigPi (formerly TinyPi). Use these names when discussing seams and
deepening opportunities so future architecture reviews share one vocabulary.

## Domain terms

- **Goal resolution** — `src/agent/goal-resolution.ts`. Stateless module that infers the current user goal from the conversation state (summary `## Goal`, exploration-ledger findings, and recent messages), and collapses continuation inputs (`continue` / `继续` / `go on` / …) back onto the previous real goal. Kept free of `ConversationContext` so it is testable in isolation.
- **Conversation context** — `src/agent/context.ts` (`ConversationContext`). Owns the summary, recent messages, entry stream, and exploration ledger; the source of truth for working context.
- **Context budget** — the token budget that drives compaction, defined **per model** (ADR 0021). Three model-level fields: `hard_context_limit` (the model's physical total token ceiling, default `200_000`), `reserveTokens` (headroom kept free so the next request fits; the soft limit is derived as `hard_context_limit - reserveTokens`), and `keepRecentTokens` (how much of the recent tail stays verbatim). `ConversationContext` reads the budget from a getter bound to the *active* model, so `/model switch` updates the compaction trigger. Retired term: `contextWindow` (agent-level) — replaced by `hard_context_limit`.
- **Compaction** — summarizing older recent messages into the summary when the token budget is exceeded (token trigger) or on demand (force).
- **Exploration ledger** — `src/agent/exploration-ledger.ts`. Structured record of what the agent has searched, read, modified, and found, injected into the working context to curb re-exploration.
- **Tool** — a capability the agent can call, defined by `ToolDefinition` and dispatched through `ToolRegistry`. Built-ins live in `src/tools/builtin/`.
- **Session** — a persisted conversation (`src/session/store.ts`). Stores the entry stream (message + compaction entries), turn history, and a recovery snapshot.
- **EntryStreamSerializer** — `src/session/format.ts`. The single module that converts between a context state (`{summary, recentMessages}`) and the entry stream in both directions: `buildEntriesFromContextState` (state → entries, from scratch), `resolveEntriesForPersist` (state → entries, append-only merge into an existing transcript), and `deriveContextStateFromEntries` (entries → state). Context and store both delegate here, so entry derivation lives in one place.
- **Skill** — an instruction document discovered from skill roots and surfaced to the system prompt (`src/skills/`). Discovery implies trust: a discovered skill is loaded and trusted with no separate gate (ADR 0017). **Refined by ADR 0022:** load-implies-trust now applies only within a *trusted project* — see **Project trust**. "Trusted skill" / "untrusted skill" / "force-load" are retired terms — under load-implies-trust there is no untrusted-discovered skill; pointing `read` at a `SKILL.md` path is just reading a file.
- **Skill root** — a skill discovery directory (`.sigpi/skills` / `.agents/skills`, project or global; enumerated by `collectSkillRoots`, `src/skills/catalog.ts:69`). **Read-only invariant removed (ADR 0023):** SigPi no longer blocks writes into skill roots — the guard was bypassable without a sandbox (e.g. `npx`, subshells, `curl | sh`) and gave false safety. Loading of *project* skill roots is instead gated by **Project trust** (ADR 0022); global skill roots are user-installed and never gated. _Avoid_: trusted roots, allowed roots — the `allowedRoots` read/write allow-list (ADR 0016) was also removed by ADR 0022.
- **Project trust** — a per-directory decision, recorded in `~/.sigpi/trust.json` (canonical absolute dir → `always` | `never`), that gates whether SigPi *loads project-local resources* before working in a directory. Trusted resources are the project's skills (`.sigpi/skills` / `.agents/skills`, walked up from cwd) and the project `.sigpi/config.toml` override; `AGENTS.md` / `CONTEXT.md` load regardless, and global skill roots are never gated (user-installed, trusted by definition). Lookup walks cwd → filesystem root and the closest saved decision wins; absent any, it falls back to the global `defaultProjectTrust` (default `ask`). In the REPL it prompts; in headless one-shot it ignores project resources (with a stderr warning). `--approve` / `--no-approve` override per run. Mirrors Pi's project trust: it guards resource *loading*, not what tools the model may call once running (ADR 0022). _Avoid_: permission, sandbox, skill-root invariant.
- **Model provider** — the `ModelProvider` seam, defined and constructed in `src/model/provider.ts` (the model module's entry point), that turns a request into a model response. Consumers obtain one via `createModelProvider(config, logger)` and never name the concrete class; `OpenAICompatibleProvider` is the thin composer over a format-agnostic `ModelTransport` and a `WireFormatAdapter` (chat-completions or responses), selecting the adapter by `config.apiFormat`.
- **Model transport** — `src/model/transport.ts`. Owns HTTP resilience shared by every wire format: fetch, generic SSE framing, stream body reading, an idle/stall timeout that resets on every received chunk, abort merging, error classification (`ModelRequestError` + `RequestFailureKind`), JSON parsing, and the retry/backoff loop. It is format-agnostic — it only parses SSE frames and hands each `data:` payload to the adapter; it never interprets delta shapes.
- **Idle/stall timeout (静默超时)** — a model request's timeout is measured as *silence*, not total duration: one timer starts at `fetch` and resets on every received SSE frame (or body chunk), firing only after `timeoutMs` with no bytes. It therefore guards both a dead server (never sends a first frame) and a mid-stream stall, while no longer killing responses that stream steadily but run long in total. _Avoid_: total-duration timeout, hard turn deadline. **Residual gap (ADR 0020, accepted for now)**: the reset fires on *any* received byte, including reasoning/thinking tokens, so a model that streams thinking indefinitely but never emits content will never trip the idle timer. This is **accepted** for the streaming-render work: once reasoning is rendered live and ESC interrupt is fixed (so a live-but-slow stream is visible and manually cancellable), the "mysterious stall" symptom is gone, and a separate `reasoning_timeout` timer is a deferred enhancement, not a required fix. The idle timer's job stays "catch dead streams", not "bound thinking duration".
- **Reasoning token (思考 token)** — model output that is not the final answer but the model's chain-of-thought, carried in a separate delta field (`reasoning_content` / `reasoning`) rather than `content`. The chat-completions adapter historically ignored it; the streaming-render work captures it so the thinking phase is visible instead of a blank `thinking` label. Distinct from **content token** (the answer text in `content`).
- **Streaming render (流式渲染)** — displaying model output incrementally as SSE frames arrives, rather than buffering until `[DONE]`. The transport/adapter/runner path is **fully buffered today**: `readSseStream` only `accumulate`s frames and returns one complete `ModelResponse` on `[DONE]`; the runner `await`s that full response before any rendering, and the only mid-flight signal is the `model_request_started` / `model_request_finished` progress events. The planned `onDelta` callback chain (`generate` → `performRequest` → `readSseStream` → runner → `TurnProgressEvent`) will carry `{reasoningDelta, contentDelta}` per frame. **Interface decision (ADR 0020)**: the `onDelta` payload carries *both* deltas from day one even though the UI's first version renders only reasoning, so adding content streaming later needs no transport/runner change. Rendering is display-only — it does not alter the agent turn control flow (still `await` full response before tool calls / compaction). **Event division (ADR 0020)**: the new `model_delta` event is *render-only*; the existing phase events (`assistant_message`, `tool_calls_received`, `tool_execution_*`) remain *persistence/logging-only* and are untouched. Tool-call events fire only after `generate()` returns the full `ModelResponse`, so they never overlap with in-flight `model_delta` frames. The two signal families are orthogonal and decoupled — `model_delta` drives the live text panel, phase events drive logs and final state. **UI shape (ADR 0020)**: the reasoning stream renders in a dedicated `ReasoningStreamComponent` added as a `Tui` *child* above the prompt and below the (bottom) status bar — never below the status bar, which is the fixed footer. The component is dim-colored and left-indented to visually distinguish reasoning from the later answer text (reserving that distinction for the content-streaming phase). Height is elastic but capped at ~70% of the content area; overflow scrolls internally so the prompt line stays visible and ESC remains reachable. **Live-preview lifecycle (ADR 0020)**: the `ReasoningStreamComponent` is a *temporary preview* driven only by in-flight `model_delta` frames. On `model_request_finished` / `assistant_message` (normal completion), on `turn_interrupted` (ESC), and on `turn_failed` / `truncated` (failure), the component is cleared and stops updating; the final whole-text display is owned by `assistant_message` (persistence/logging event, per the event-division decision). This avoids double-printing the reasoning (live + whole) and leaves no half-rendered residue on interrupt/truncation; the failed/truncated diagnostic reasoning print (non-TTY case) is the only place reasoning appears outside the live component. **Non-TTY / one-shot modes (ADR 0020)**: `model_delta` is silently dropped in `console.log`-based renderers (compact/clear/quiet) — stdout is linear and per-frame printing would spam and is meaningless when piped. The final answer still prints whole via `assistant_message`. Exception: on a *failed/truncated* turn, the reasoning accumulated so far is printed as diagnostic output, because in the `finish_reason: "length"` case the user most needs to see what the model was thinking.
- **Interrupt vs retry (中断 vs 重试)** — a user interrupt (ESC → `interruptController.requestInterrupt()` → aborted `AbortSignal`) must be treated as a terminal cancellation, **never** as a retryable network error. **Bug fixed by ADR 0020**: undici's `fetch` surfaces an abort as its own `AbortError` DOMException and does *not* re-throw `signal.reason` (`TurnInterruptedError`), so the transport's catch must detect `AbortError` + `request.abortSignal?.aborted` and throw the interrupt error directly — not classify it as `aborted`/`network_error` and let `isRetryableRequestError` retry it. Before the fix, ESC aborted the in-flight request, the retry loop re-issued it, and the turn re-entered `thinking`, making ESC appear to "do nothing".
- **Wire format adapter** — `src/model/wire-format.ts` (`WireFormatAdapter`) with two implementations: `ChatCompletionsAdapter` and `ResponsesAdapter`. Each owns its endpoint (`buildUrl`), request-body shape (`toRequestBody`), single-shot response parsing (`parse` — used on the non-streaming path and the single-JSON fallback), and **streaming delta accumulation**: `accumulate(frame)` folds one SSE `data:` payload into running state and `finalize()` emits the complete `ModelResponse`. Keeping delta-shape knowledge in the adapter is what lets the transport stay format-agnostic. Adding an API format means adding one adapter, not forking the transport.
- **Agent turn** — one `runTurn()` pass in `src/agent/runner.ts`: user input → some number of tool-call steps → final answer. This is the concrete shape of the "agent loop" in this project; step count is bounded by `maxSteps`. Exceeding the bound ends the turn with a **local** fallback answer built from in-memory `toolExecutions` + `currentGoal` (enhanced `buildMaxStepsFallbackAnswer`) — **no final `tools: []` synthesis call** (that call leaked `<tool_call>` markup and added 500 risk; removed per ADR 0018). The fallback is user-facing only: it states the limit was reached, the task is incomplete, and prompts `go on` to continue.
- **Max-steps fallback** — the local answer emitted when `step > maxSteps`. Assembled from real `toolExecutions` (files read, commands run, calls made) + `currentGoal`; never from a model call. Distinct from **turn checkpoint** (a compaction summary) and from **tool error** (surfaced into a completed turn, `ok: true`, not a fallback).
- **Resume-on-"go on"** — when a turn ends via the `maxSteps` bound, it is marked resumable; a later `go on` (`goal-resolution.ts` collapses `continue`/`继续`/`go on`/`resume` onto the previous goal) **resumes the same task from the checkpoint** with a fresh `maxSteps` budget, rather than a cold restart that re-runs the same steps (ADR 0018). The "reached max steps" signal is never injected into a message to the LLM — the next turn resumes from real history + checkpoint.
- **Error exit contract** — model errors make the CLI exit non-zero (`runAsk` maps `RunTurnResult.ok === false` → `process.exitCode = 1`); tool errors and the max-steps fallback exit 0 (ADR 0018). `ok: false` already means "no answer produced" (only model errors reach it), so the contract is encoded in the type.
- **Function calling** — the mechanism by which the model emits `toolCalls` in its response, the runner dispatches them via `ToolRegistry`, and the results are fed back into the working context. Distinct from **Tool** (the capability definition): a Tool is "what can be done"; function calling is "how the model drives it".
- **Context management** — see **Compaction**; this project keeps the working context within the token budget using a three-part scheme of "summary + recent messages + in-turn checkpoint", corresponding to one of the three teaching concepts claimed by the README.
- **Process output mode** — the `[agent] process_output` config value that selects how the CLI renders a turn's progress. `compact`: dense, Claude-Code-style tier showing the user + assistant messages, grouping parallel tool calls returned in one model response, with reduced tool results. `detailed` (default): `compact` plus turn/model-run dividers, counts, and fuller tool-result rendering. _Avoid_: quiet, clear, full — prior names; `full` was removed.
- **Command suggestion** — a `ChatCommandMetadata` whose name matches the current single-`/token` buffer (`/^\/\S*$/`), surfaced below the input line in `ChatInputComponent`. Computed by `getChatCommandSuggestions`; includes dynamic `/skill:<name>` entries.
- **Selected suggestion** — the suggestion at `selectedSuggestionIndex` (default 0), rendered with the `> ` marker; the target of both Up/Down navigation and Tab-completion.
- **Tab completion** — in the `/xxx` command-entry line, Tab fills the selected suggestion's name plus a trailing space into the input buffer, replacing the current `/token`, **without submitting**. Tab is swallowed even when no suggestion exists (never inserts a literal tab). Distinct from Enter, which submits.
- **Suggestion token** — the buffer form `/^\/\S*$/` (a single slash-token with no spaces) under which suggestions are computed; completion applies only to this token, not to post-space arguments (e.g. `/model <id>` argument completion is out of scope).
- **Input history (命令历史)** — shell-style recall of previously submitted inputs via `↑`/`↓` in the chat input. **In-memory and process-scoped**: a single global buffer for the CLI run, discarded on exit; not persisted to disk and not tied to a `Session`. Coexists with command-suggestion navigation by **context**: when the current line is a `/` chat command with suggestions open, `↑`/`↓` navigate suggestions (unchanged behavior); otherwise `↑`/`↓` recall history. Recall loads a prior input into the editor for editing, preserving the in-progress draft when scrolling past either end. **Entries**: only inputs that actually reach the model are recorded — natural-language prompts and model-driving commands such as `/skill:…` — with consecutive-duplicate suppression. Local-only commands that never send to the model (`/settings`, `/models`, `/help`, …) are **not** recorded and are not recallable, mirroring Pi's behavior. **Navigation**: the live draft is a distinct slot at the bottom of the list; `↓` past the newest entry returns to the draft; `↑` past the oldest entry stops (no wrap). The suggestion-vs-history context rule is **stateless**: arrows navigate suggestions exactly when `getChatCommandSuggestions` would return non-empty (buffer matches `/^\/\S*$/`), otherwise they recall history — no extra "recalling" mode flag. **Multiline**: recalled entries may contain newlines and load whole into the editor. Arrow-meaning is tied to history position — on a history entry (or the draft) `↑`/`↓` recall; once the recalled line is edited it drops back to the draft slot and `↑`/`↓` become in-editor vertical cursor movement. **Ownership & write path**: a single `InputHistory` buffer is created once at CLI startup and injected into the REPL loop. Both the idle input and the running-turn input converge on `executeChatCommand` in the loop; the buffer is pushed exactly when the command result's `turnInput` is non-null (i.e. the input reached the model), recording the **original `line`** the user typed (not any expanded form such as a `/skill:` instruction). The running-turn input shares the same buffer and the same recall/record rules, so history behaves identically in both modes. The `Editor` stays a pure text-editing primitive and history navigation is a component-level concern (so it can interact with the suggestion-nav context). Distinct from **Session** persistence — the conversation transcript already preserves what was said; history is only about re-typing what you typed, and is intentionally not durable. See ADR 0018.

## Conventions

- **One-shot prompts persist a session by default.** A `sigpi "prompt"` with no flags creates and persists a Session (mirroring `claude -p`, `codex exec`, and Pi, which all keep a session by default). Ephemeral, fire-and-forget execution is the explicit exception, reached via a `--no-session` opt-out — not the default. Consequence for the architecture: every `Agent turn` is a Session turn, so the runtime need not keep a parallel bare `AgentRunner` alongside `SessionRuntime`; the opt-out path is the only place a non-persisted runner is constructed.

## Status bar

The fixed footer line above the prompt showing live session state. Rendered by `formatStatusBar` / `formatStatusBarForEvent` in `src/chat-repl.ts`; the git branch lookup lives in `src/git.ts`. See ADR 0022 for the full rationale.

- **Composition** — three segments joined by ` | `: `tokens {used}/{limit} ({pct}%) [Hit({pct}%)] | {cwd} ({branch})`. The cache hit rate is appended inline to the token segment (no `|` separator) because it's a property of the same response. The `model {modelName}` prefix was removed (the model is already visible in the prompt area and via `/model`).
- **Token segment — before the first model response** — when `ConversationContext.getLastUsage()` is `null`, renders `tokens ?/{limit}` with **no percentage**. The limit is known from the active model's `hard_context_limit`; the used count is unknown, so it is shown as `?` rather than falling back to the `chars/4` heuristic (which is inaccurate, especially for CJK and code).
- **Token segment — after the first response** — renders `tokens {lastUsage.totalTokens}/{limit} ({pct}%)`. The provider's `totalTokens` is ground truth; the display stays at that value until the next response lands. **No trailing-delta estimate** is shown when the user types a follow-up — staleness is honest, a wrong estimate is not.
- **Cache hit rate segment** — appended inline to the token segment (no `|` separator). Formula: `cacheRead / (cacheRead + input) * 100`, one decimal place. Rendered as `Hit({pct}%)`. Hidden when `lastUsage` is `null` (same lifecycle as the token segment) or when `cacheRead + input === 0` (no cacheable input, or provider doesn't report cache fields). A legitimate 0% hit rate (`cacheRead === 0 && input > 0`) renders as `Hit(0.0%)`. Examples: `tokens 12.3K/183.6K (7%) Hit(85.3%) | ~/repo (main)`, `tokens 12.3K/183.6K (7%) Hit(0.0%) | ~/repo (main)`, `tokens 12.3K/183.6K (7%) | ~/repo (main)` (no cache segment).
- **Token segment — after `ConversationContext.recover()`** — `recover()` clears `lastUsage` (the recovered tail is a different conversation), so the segment drops back to `tokens ?/{limit}` until the next response.
- **Token segment — on session resume** — `lastUsage` is persisted on the assistant `MessageEntry` (optional `usage?: ModelUsage` field). `hydrateState` scans `recentMessages` for the last assistant message with `usage !== undefined` and sets `lastUsage = { usage, messageIndex: that index }`. So a resumed session shows the real number from the last response, not `?`. Old sessions (no `usage` field) → scan finds nothing → `lastUsage = null` → `?` (no regression). `usage` is persisted regardless of turn outcome — `length` truncations and interrupts are the most useful cases (they tell the user "you hit the limit").
- **CWD segment** — `shortenWorkingDirectory(getCurrentWorkingDirectory(state))`, e.g. `~/repo`.
- **Branch segment** — appended to cwd in parens. Normal: `~/repo (main)`. Detached HEAD: `~/repo (@a1b2c3d)` (short SHA prefixed with `@`). Not-a-repo / timeout / error: segment omitted silently, e.g. `~/repo`.
- **Git lookup** — `getGitBranch(cwd)` in `src/git.ts` spawns `git rev-parse --abbrev-ref HEAD` with a 50ms timeout; on `HEAD` (detached) it falls back to `git rev-parse --short HEAD` and prefixes with `@`. Module-level `Map<string, string>` cache keyed by `cwd`, no TTL. Returns `null` on any failure; never throws. Never blocks the prompt.
- **Async** — `formatStatusBar` and `formatStatusBarForEvent` are `async` (the git lookup is async). Call sites in `src/cli.ts` (`readInput`, `startRunningTurnInputListener`) `await` the result.

## Seam artifacts

- **Tool seam** — the `ToolDefinition` interface. Deepened so each tool may carry its own progress/ledger adapters instead of tool-specific code living elsewhere.
  - `describeProgress(args)` (optional) — returns `{ summary; detail? }` for progress reporting.
  - `recordLedger(recorder, toolCall, result)` (optional) — records exploration into the ledger via the `LedgerRecorder` facade.
- **LedgerRecorder** — a facade passed to a tool's `recordLedger` adapter. Exposes intent methods (`search`, `read`, `modified`, `finding`, `rejected`, `shellFinding`); the ledger module implements it and owns caps, dedup, and normalization. Keeps ledger invariants out of the tool definitions.
- **ledgerRecorder callback** — injected into `ConversationContext` (mirrors `compactionHooks`) so the context can record into the ledger without depending on the tool registry.
